## 一、我们最初的规划 vs 实际发生的

### 最初设想

```
grill-me 脑暴 idea → 需求对齐
    →
frontend-design 控制 UI 质量 → Claude 自由发挥代码水平
    →
我测试验收 → 提 bug → Claude 修复 → 循环迭代
```

这个流程本身没问题，问题出在**执行环节**。

### 实际发生了什么

1. **grill-me 只走了一次**：在项目开头脑暴了 MVP 功能池，但后续每个具体功能（侧边栏改造、拖拽、右键菜单）都没有重新 grill，需求是我一句话描述后就写了
2. **frontend-design 没被有效利用**：装了但写代码时没有主动参照它的设计标准，导致第一版 macOS 风格还是偏"我自己编的"，用户指出后才认真按照 skill 重新设计
3. **缺少自检环节**：写完代码直接交付，没有 `npm run build` 验证、没有 API 端到端测试，导致多次编译错误和运行时 bug

## 二、优化后的开发流程（从现在开始执行）

```
┌─────────────────────────────────────────────────────────────┐
│  阶段 0 · 需求对齐                                          │
│  - 用户描述需求                                             │
│  - Claude 追问细节（模拟 grill-me），直到验收标准清晰        │
│  - 产出：3-5 条验收标准，写入 md 文档                        │
├─────────────────────────────────────────────────────────────┤
│  阶段 1 · 设计方案（仅 UI 相关）                             │
│  - Claude 基于 frontend-design 技能，输出设计方案：          │
│    字体选择、配色、间距、动效策略                            │
│  - 用户确认后再写代码                                       │
│  - 产出：设计卡片，写入 md 文档                              │
├─────────────────────────────────────────────────────────────┤
│  阶段 2 · 编码实现                                          │
│  - Claude 写代码                                            │
│  - 遵循 vercel-react-best-practices 性能规则                 │
├─────────────────────────────────────────────────────────────┤
│  阶段 3 · 自检（Claude 主动执行，不可跳过）                  │
│  - npm run build → 确保 0 编译错误                          │
│  - curl 测试关键 API → 确保运行时可用                       │
│  - 对照验收标准逐条检查                                      │
│  - 产出：自检结果，写入终端输出 + md 文档                    │
├─────────────────────────────────────────────────────────────┤
│  阶段 4 · 交付 → 用户测试                                   │
│  - Claude 汇报"做了什么 + 怎么测的"                         │
│  - 用户测试并提出 bug                                       │
│  - Claude 修复 → 重新自检 → 交付                            │
└─────────────────────────────────────────────────────────────┘
```

### 和之前的区别

| 维度 | 之前 | 之后 |
|---|---|---|
| 需求对齐 | 一句话就写 | 写出验收标准再动手 |
| 设计方案 | 写完再改 | UI 先定调，确认后再写 |
| 自检 | 无 | `build + API test` 不可跳过 |
| 文档 | 事后补 | 每个阶段的产出实时写 |
| 技能使用 |装了没用| frontend-design 在阶段1强制介入|

# 功能：全文批注系统

> 状态：规划完成 / 待开发
> 创建日期：2026-06-06
> 关联功能：[[页面永久存档]] [[学习路径]] [[AI知识问答(RAG)]] [[书签详情]]

---

## 一、概述

为 MarkBox 新增「全文批注系统」——在阅读模式中渲染书签的正文内容（基于 Mozilla Readability 提取），用户可直接在原文上高亮划线并写旁注。支持多色高亮、旁注编辑、批注导出 Markdown。对标 Cubox/Omnivore/Matter 的核心阅读批注体验，将 MarkBox 从"书签整理工具"升级为"阅读内化工具"。

---

## 二、技术选型

### 决策 1：批注载体

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 在书签详情面板中批注（BookmarkDetailSheet 底部增加笔记区块） | 融入现有界面，开发量小 | 看不到原文内容，批注缺乏上下文锚点——"我划这句话是因为前面说了什么？" |
| **B** | 在阅读模式中批注 — 新增阅读模式视图，渲染提取的正文，用户直接在原文上划线+写旁注 | 沉浸式阅读+批注一体，对标 Matter/Omnivore 的核心体验 | 需要 Feature C（页面存档）的抓取能力作为前置依赖。阅读模式渲染需要额外开发 |
| **C** | 两者都支持 — 详情面板笔记 + 阅读模式批注 | 最灵活覆盖所有场景 | 开发量翻倍，两套批注 UI 需要统一数据模型 |

> ✅ **推荐 B**。阅读模式中批注是 Cubox/Omnivore/Matter 的核心体验，也是用户最自然的批注场景——"边读边划"。以 Mozilla Readability 提取的正文作为阅读内容源，以存档 HTML 作为降级方案。

> ❌ **排除 A**：看不到原文的批注缺乏上下文锚点。详情面板适合"元信息浏览"，不适合"沉浸式阅读+批注"。
>
> ❌ **排除 C**：两套批注系统在 MVP 阶段过度设计。阅读模式批注已覆盖核心场景，详情面板保留现有 AI 摘要视图即可。

---

### 决策 2：批注类型

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 仅文本笔记 — 在书签上写自由文本笔记 | 实现简单 | 笔记与原文无位置关联——"关于这篇文章的感想"而非"关于这段话的批注" |
| **B** | 高亮划线 + 旁注 — 选中原文文字 → 高亮标记 → 可附加旁注 | 批注与原文位置绑定，最自然的阅读批注体验（Medium/Cubox 模式） | 需要在 HTML 渲染中实现选区捕获和文本位置锚定（XPath + 偏移量） |
| **C** | B + 图片区域标注 + 语音批注 | 覆盖多媒体批注场景 | 图片标注和语音批注大幅增加复杂度，MVP 阶段性价比低 |

> ✅ **推荐 B**。高亮划线 + 旁注是阅读批注的核心交互范式，已被 Medium、Cubox、Omnivore 验证。高亮支持 4 种颜色语义：
> - 🟡 黄色 = 重点/关键信息
> - 🟢 绿色 = 可引用/好观点
> - 🔵 蓝色 = 疑问/待深入
> - 🔴 红色 = 反对/需验证

> ❌ **排除 A**：纯文本笔记与原文无位置关联，不是真正的"批注"系统，学习路径的 PathNote 已覆盖此场景。
>
> ❌ **排除 C**：图片标注需要 Canvas 叠加层，语音批注需要录音+存储+播放，MVP 阶段过度设计。

---

### 决策 3：批注存储与导出

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 仅服务端存储（Prisma DB） | 实现简单，与现有架构一致，跨设备同步 | 用户离线时无法查看已有批注 |
| **B** | 服务端存储 + 导出 Markdown/JSON | 批注不锁定在平台内，用户可带走 | 导出格式设计需要额外工作 |
| **C** | B + 浏览器扩展本地缓存（离线可用） | 离线可用 | 本地-服务端同步冲突处理复杂 |

> ✅ **推荐 B**。服务端存储保证跨设备同步。导出为 Markdown 让用户可以将批注带走——"数据属于用户"。Markdown 导出格式：`> 原文高亮内容\n\n💡 我的旁注`。与学习路径导出一致。

> ❌ **排除 A**：批注是用户的知识产出，锁定在平台内是反模式。导出能力建立用户信任。
>
> ❌ **排除 C**：离线同步在 MVP 阶段引入冲突处理复杂度，性价比不足。

---

### 决策 4：批注与学习路径的联动

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 独立使用 — MVP 阶段批注系统与学习路径不联动 | 概念清晰，两个功能独立迭代 | 批注和路径是两个孤岛 |
| **B** | 路径节点可引用批注 — 学习路径的 PathNote 可引用/嵌入某篇书签的批注 | 批注和路径关联——"读到这篇时，之前划的重点自动出现" | 跨模型引用，UI 复杂度增加 |
| **C** | 批注自动归入相关学习路径 | AI 自动关联 | 自动归入可能不准确，需用户确认，反而增加操作步骤 |

> ✅ **推荐 A**（MVP）。批注系统定位"深度阅读记录"，学习路径定位"学习进度追踪"。两者场景不同，MVP 阶段保持独立，后续根据用户反馈决定联动方式。

> ❌ **排除 B**：跨模型引用增加 Prisma 查询和 UI 复杂度，MVP 阶段不必要。
>
> ❌ **排除 C**：自动归入准确性无法保证，错误归入造成的困扰大于便利。

---

### 决策 5：阅读模式渲染方案

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 使用存档 HTML 直接渲染（Feature C 存档数据） | 与存档功能复用数据，零额外抓取 | 简化 HTML 可能残留非正文内容，DOM 结构复杂，批注锚定易失效 |
| **B** | Mozilla Readability 提取正文 + 自定义阅读样式 | 阅读体验好——清晰排版、去杂质。DOM 简单，批注锚定可靠 | 需要引入 Readability 库 |
| **C** | A + B 混合 — 优先 Readability，失败时降级到存档 HTML | 最佳体验+兜底方案 | 两套渲染路径需要统一适配层 |

> ✅ **推荐 B**。Mozilla Readability 是 Firefox 阅读模式的核心引擎，已被 Omnivore 验证。Readability 提取失败时降级使用存档 HTML 渲染（作为 fallback，非并列选项）。

> ❌ **排除 A**：简化 HTML 可能残留导航栏、侧栏等非正文元素，且 DOM 复杂导致批注位置锚定不可靠。
>
> ❌ **排除 C**：两套同等级渲染引擎维护成本翻倍。Readability 为主 + 存档降级即可。

## 三、交互设计

### 3.1 阅读模式

```
进入阅读模式（从书签详情点击「阅读」按钮或从列表右键菜单）
  → 全屏/居中阅读视图
  → 顶部工具栏：返回 / 目录（如有） / 批注列表 / 导出 / 阅读进度
  → 正文区域：Readability 提取的排版内容
  → 选中文字 → 弹出批注工具栏（4 色高亮 + 写旁注）
  → 已有高亮 → 点击查看/编辑旁注
  → 右侧可选面板：当前书签的全部批注列表
```

### 3.2 批注操作

- **高亮划线**：选中文字 → 弹出 mini 工具栏 → 选择颜色标记
- **写旁注**：点击高亮区域 → 弹出旁注输入框 → 保存
- **查看批注**：点击高亮区域 → 显示旁注卡片
- **删除批注**：点击高亮区域 → 「删除」按钮
- **批注列表面板**：按颜色筛选、按位置排序

### 3.3 批注导出

导出 Markdown 格式：
```markdown
# 批注：《Rust 异步编程指南》
> 导出时间：2026-06-06
> 共 12 条批注（🟡 5 · 🟢 3 · 🔵 2 · 🔴 2）

## 🟡 重点
1. > async/await 是 Rust 异步编程的核心语法糖...
   💡 这里解释得很清楚，async 本质是返回 impl Future

2. > Future trait 只有一个方法：poll...
   💡 和 JS Promise 完全不同，Rust 的 Future 是惰性的
...
```

## 四、技术实现

### 4.1 数据模型

```prisma
model Annotation {
  id          String   @id @default(cuid())
  userId      String
  bookmarkId  String
  type        String   // highlight | note
  color       String?  // yellow | green | blue | red (仅 highlight)
  text        String   // 被选中的原文
  note        String?  // 旁注内容
  // 位置锚定
  anchor      String   // JSON: { xpath, startOffset, endOffset, textPrefix, textSuffix }
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  user        User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  bookmark    Bookmark @relation(fields: [bookmarkId], references: [id], onDelete: Cascade)

  @@index([userId, bookmarkId])
}
```

### 4.2 位置锚定机制

使用 Web Annotation 标准的 XPath + 文本偏移量方案（被 Omnivore/Cubox 验证）：

```typescript
interface AnnotationAnchor {
  xpath: string;          // 选中起点的 DOM XPath
  startOffset: number;    // 起点在文本节点中的偏移
  endOffset: number;      // 终点在文本节点中的偏移
  textPrefix: string;     // 锚点前 20 字符（用于模糊匹配恢复）
  textSuffix: string;     // 锚点后 20 字符
  selectedText: string;   // 完整选中文本
}
```

- **定位**：读取时优先用 XPath+偏移量精确定位，失败时降级为文本前缀/后缀模糊匹配
- **鲁棒性**：即使页面 HTML 结构有微小变化，文本前后缀匹配仍能恢复批注位置

### 4.3 API 设计

| 端点 | 方法 | 说明 |
|---|---|---|
| `/api/bookmarks/[id]/annotations` | GET | 获取书签的所有批注 |
| `/api/bookmarks/[id]/annotations` | POST | 创建批注 |
| `/api/bookmarks/[id]/annotations/[annotationId]` | PATCH | 更新批注（修改颜色/旁注） |
| `/api/bookmarks/[id]/annotations/[annotationId]` | DELETE | 删除批注 |
| `/api/bookmarks/[id]/annotations/export` | GET | 导出批注为 Markdown |
| `/api/bookmarks/[id]/readable` | GET | 获取 Readability 提取的正文 |

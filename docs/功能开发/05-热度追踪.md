# 功能：书签热度追踪

> 状态：规划完成 / 待开发
> 创建日期：2026-06-05
> 关联功能：[[AI自动分类]] [[仪表盘]] [[浏览器扩展]] [[时间线视图]]

---

## 一、概述

为 MarkBox 新增「热度追踪」功能，让收藏的书签不再是静态快照，而是持续追踪其热度变化。从 GitHub 仓库的 star/fork 增长、B 站视频的播放量变化，到普通网页的标题/描述变更和链接失效检测。热度数据以时间序列存储，通过迷你折线图和 AI 预测趋势线在 Dashboard 和独立动态页面中展示。重要里程碑突破时发送通知。

书签从「存了就忘」变成「活的资产」——用户能看到自己收藏的内容如何随时间演变，这是答辩的核心差异化亮点之一。

## 二、技术选型

| 决策点 | 选择 | 理由 |
|---|---|---|
| **追踪数据来源** | C：GitHub（优先）→ 社交媒体（B 站/知乎）→ 通用网页变化检测（分层覆盖） | GitHub API 最标准化、技术书签占比高、API 完善。社交媒体按需扩展，通用网页变化检测做兜底。答辩展示从结构化到非结构化的完整追踪体系 |
| **更新频率** | B：智能自适应频率 | GitHub 活跃仓库每小时检查，低频仓库每天；B 站新视频前 3 天高频、后低频；普通网页每天。用户可手动触发即时检查。全局速率限制保护外部 API |
| **通知机制** | C：Dashboard 卡片（轻量） + 独立动态页（详细） + 邮件/浏览器推送（重要里程碑） | 三层递进——日常变化不打扰（Dashboard 即可看到），深度浏览有专门页面，里程碑突破才推送通知 |
| **数据存储** | B：时间序列快照 + 迷你折线图（sparkline） + AI 趋势预测 | 每次检查存时间戳 + 数值，用 Recharts 渲染变化曲线。AI 线性回归预测未来趋势（标注"仅供参考"） |
| **GitHub API** | REST API v3，可选 Personal Access Token 提升速率限制 | 未认证 60 次/小时，认证后 5000 次/小时。引导用户在设置中配置 Token |
| **社交媒体数据** | 服务端 HTML 解析（cheerio，已有依赖）+ 平台特定规则 | 避免依赖第三方爬虫 API。B 站/知乎页面数据在 HTML 中可读取（OG 标签 + JSON-LD + 内嵌数据） |

```typescript
// src/lib/tracking/github.ts

interface GitHubMetrics {
  stars: number;
  forks: number;
  openIssues: number;
  latestRelease: {
    tag: string;
    publishedAt: string;
  } | null;
  lastCommitAt: string;
  language: string;
  license: string | null;
  archived: boolean;
}

async function fetchGitHubMetrics(repoUrl: string): Promise<GitHubMetrics> {
  // 1. 解析 GitHub URL → owner/repo
  const { owner, repo } = parseGitHubUrl(repoUrl);
  
  // 2. 并行获取仓库信息和最近 commit
  const [repoData, commits, releases] = await Promise.all([
    githubAPI(`/repos/${owner}/${repo}`),
    githubAPI(`/repos/${owner}/${repo}/commits?per_page=1`),
    githubAPI(`/repos/${owner}/${repo}/releases?per_page=1`),
  ]);
  
  return {
    stars: repoData.stargazers_count,
    forks: repoData.forks_count,
    openIssues: repoData.open_issues_count,
    latestRelease: releases.length > 0 ? {
      tag: releases[0].tag_name,
      publishedAt: releases[0].published_at,
    } : null,
    lastCommitAt: commits[0]?.commit?.author?.date || repoData.pushed_at,
    language: repoData.language || 'Unknown',
    license: repoData.license?.spdx_id || null,
    archived: repoData.archived,
  };
}
```

### 模块 B：智能调度引擎

#### B.1 自适应频率策略

```typescript
// src/lib/tracking/scheduler.ts

interface TrackingSchedule {
  bookmarkId: string;
  sourceType: 'github' | 'bilibili' | 'zhihu' | 'webpage';
  interval: number;        // 当前检查间隔（毫秒）
  lastCheckedAt: string;
  lastChangeAt: string | null;
  changeFrequency: 'high' | 'medium' | 'low' | 'dormant';
}

function calculateNextInterval(schedule: TrackingSchedule): number {
  const now = Date.now();
  const daysSinceCreation = (now - new Date(schedule.bookmark.createdAt).getTime()) / 86400000;
  const daysSinceLastChange = schedule.lastChangeAt
    ? (now - new Date(schedule.lastChangeAt).getTime()) / 86400000
    : Infinity;
  
  switch (schedule.sourceType) {
    case 'github':
      // 活跃（7 天内有 commit）→ 6h；一般 → 12h；不活跃 → 24h
      if (schedule.changeFrequency === 'high') return 6 * 3600000;
      if (schedule.changeFrequency === 'medium') return 12 * 3600000;
      return 24 * 3600000;
    
    case 'bilibili':
      // 发布 3 天内 → 6h；7 天内 → 12h；30 天内 → 24h；之后 → 48h
      if (daysSinceCreation <= 3) return 6 * 3600000;
      if (daysSinceCreation <= 7) return 12 * 3600000;
      if (daysSinceCreation <= 30) return 24 * 3600000;
      return 48 * 3600000;
    
    case 'webpage':
      // 普通网页 → 24h（每天检查一次变化和失效）
      return 24 * 3600000;
    
    default:
      return 24 * 3600000;
  }
}
```

#### B.2 全局速率限制

```typescript
// 全局速率限制器
const RATE_LIMITS = {
  github: {
    unauthenticated: { maxPerHour: 60, maxPerMinute: 10 },
    authenticated: { maxPerHour: 5000, maxPerMinute: 80 },
  },
  bilibili: { maxPerHour: 120, maxPerMinute: 10 },  // 避免被反爬
  webpage: { maxPerHour: 300, maxPerMinute: 30 },
};

// 优先级队列：活跃仓库 > 新内容 > 普通网页
function getCheckPriority(schedule: TrackingSchedule): number {
  let priority = 0;
  if (schedule.changeFrequency === 'high') priority += 100;
  if (schedule.changeFrequency === 'medium') priority += 50;
  // 距上次检查越久，优先级越高
  const hoursSinceLastCheck = (Date.now() - new Date(schedule.lastCheckedAt).getTime()) / 3600000;
  priority += Math.min(hoursSinceLastCheck, 48) * 2; // 最多加 96 分
  return priority;
}
```

#### D.4 AI 趋势预测

使用简单的线性回归预测未来趋势：

```typescript
// src/lib/tracking/predict.ts

interface TrendPrediction {
  predictedValue: number;
  daysToMilestone: number | null;  // 距离下一个里程碑的天数
  confidence: 'high' | 'medium' | 'low';
  trend: 'rising' | 'stable' | 'declining';
}

function predictTrend(snapshots: TrackingSnapshot[]): TrendPrediction {
  if (snapshots.length < 7) {
    return { predictedValue: 0, daysToMilestone: null, confidence: 'low', trend: 'stable' };
  }
  
  // 取最近 30 天的数据点
  const recentData = snapshots
    .filter(s => s.timestamp > Date.now() - 30 * 86400000)
    .map(s => ({
      x: new Date(s.timestamp).getTime(),
      y: s.stars || s.views || 0,
    }));
  
  // 简单线性回归
  const { slope, intercept, r2 } = linearRegression(recentData);
  
  // R² < 0.5 → 趋势不可靠，置信度低
  const confidence = r2 > 0.8 ? 'high' : r2 > 0.5 ? 'medium' : 'low';
  const trend = slope > 0.01 ? 'rising' : slope < -0.01 ? 'declining' : 'stable';
  
  // 预测 30 天后的值
  const predictedValue = intercept + slope * (Date.now() + 30 * 86400000);
  
  // 计算距离下一个里程碑的天数
  const nextMilestone = findNextMilestone(recentData[recentData.length - 1].y);
  const daysToMilestone = slope > 0
    ? (nextMilestone - recentData[recentData.length - 1].y) / slope / 86400000
    : null;
  
  return { predictedValue, daysToMilestone, confidence, trend };
}
```

### 4.1 数据模型

```prisma
// 追踪调度配置
model TrackingConfig {
  id          String   @id @default(cuid())
  bookmarkId  String   @unique
  sourceType  String   // github | bilibili | zhihu | webpage
  enabled     Boolean  @default(true)
  interval    Int      @default(86400000)  // 当前检查间隔（毫秒）
  lastCheckedAt String?
  lastChangedAt String?
  changeFrequency String @default("medium") // high | medium | low | dormant
  createdAt   String   @default(now())
  
  bookmark    Bookmark @relation(fields: [bookmarkId], references: [id], onDelete: Cascade)
  snapshots   TrackingSnapshot[]
}

// 追踪快照（时间序列）
model TrackingSnapshot {
  id          String   @id @default(cuid())
  configId    String
  timestamp   String   @default(now())
  
  // 主要指标（根据 sourceType 不同字段有值）
  metricName  String?   // stars | views | upvotes | title
  metricValue Float?    // 统一数值
  metricPrev  Float?    // 上一次的值（方便计算变化）
  changePercent Float?  // 变化百分比
  
  // 完整快照数据（JSON，存储所有指标）
  snapshot    String    // JSON: TrackingSnapshotData
  
  config      TrackingConfig @relation(fields: [configId], references: [id], onDelete: Cascade)
  
  @@index([configId, timestamp])
}

// 追踪动态事件（用户可见的通知）
model TrackingEvent {
  id          String   @id @default(cuid())
  userId      String
  bookmarkId  String
  type        String   // milestone | change | alert
  title       String   // "React star 突破 5000"
  description String   // 详细描述
  data        String   // JSON: 相关数据
  isRead      Boolean  @default(false)
  createdAt   String   @default(now())
  
  user        User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  
  @@index([userId, createdAt])
}
```

**调度实现（使用已有的 CronCreate 或 Next.js Route Handler + Cron）：**

```typescript
// 定时执行（如每 5 分钟一次）
// src/app/api/tracking/cron/route.ts

export async function GET() {
  // 1. 获取所有需要检查的配置（lastCheckedAt + interval < now）
  const dueConfigs = await prisma.trackingConfig.findMany({
    where: {
      enabled: true,
      OR: [
        { lastCheckedAt: null },
        {
          lastCheckedAt: {
            lt: new Date(Date.now() - /* interval */).toISOString(),
          },
        },
      ],
    },
    orderBy: { lastCheckedAt: 'asc' }, // 最久未检查的优先
  });
  
  // 2. 按优先级排序 + 应用速率限制
  const sorted = dueConfigs.sort((a, b) => getCheckPriority(b) - getCheckPriority(a));
  
  // 3. 分批处理
  for (const config of sorted.slice(0, MAX_BATCH_SIZE)) {
    await checkAndUpdateSnapshot(config);
  }
  
  return NextResponse.json({ checked: Math.min(sorted.length, MAX_BATCH_SIZE) });
}
```

## 五、UI/UX 设计要点

### 5.1 动态卡片

- **里程碑卡片**：金色边框 + 🏆 图标，视觉冲击力强
- **常规变化卡片**：蓝色/绿色左边框，低调但信息完整
- **告警卡片**：红色左边框 + ⚠️ 图标，醒目但不恐慌
- **Sparkline**：Canvas 渲染，120×30px，无坐标轴，纯趋势形状
- **相对时间**：使用 `date-fns`（已有依赖）的 `formatDistanceToNow`

### 5.2 动画

- **动态列表加载**：卡片从下往上 staggered 出现（每条延迟 50ms）
- **数值变化**：数字跳动动画（从旧值翻转到新值，类似 odometer）
- **里程碑突破**：Dashboard 中里程碑卡片带有金色光效扫描动画
- **折线图切换时间范围**：曲线流畅过渡到新范围（motion 动画）

### 5.3 响应式

- 动态列表在移动端为单列 Feed
- Sparkline 在移动端缩小为 80×24px
- 完整图表在移动端简化（仅显示 7 天/30 天，移除 90 天选项）

### 5.4 空态与边缘情况

- **未开启任何追踪**：引导卡片——"开启热度追踪，让你的书签'活'起来"[开始设置 →]
- **追踪中但暂无变化**：显示"暂无变化" + 上次检查时间
- **被追踪页面失效（404）**：显眼的红色告警——"此页面已失效，可能已删除或迁移" [停止追踪] [手动检查]
- **GitHub 仓库归档**：蓝色提示——"此仓库已被作者归档，不再更新" [停止追踪]
- **API 限流**：黄色提示——"检查频率暂时降低，预计 X 分钟后恢复"

## 七、答辩演示脚本

**场景：展示书签的"生命力"**

> "传统的书签管理器是静态的——收藏了就放在那里，不会告诉你任何变化。
>
> MarkBox 的热度追踪功能让书签'活'了起来。比如这个我 3 个月前收藏的 React 仓库——当时它有 65,000 star，现在已经涨到 90,000。系统每个月自动抓取数据，记录下整个增长过程。
>
> 在动态页面可以看到完整的变化曲线。这里标注了 milestones——突破 1000、5000、10000。AI 还预测了未来的趋势——按这个速度，大约 15 天后会突破 100,000 star。
>
> 不只是 GitHub。B 站视频的播放量变化、知乎文章的点赞增长，甚至普通网页的标题变更和链接失效，我们都能追踪。一旦检测到重要变化——比如突破了里程碑，或者网页 404 了——系统会通过邮件或浏览器通知提醒你。
>
> 这个功能体现了 MarkBox 的核心设计哲学：书签不是终点，而是知识的起点。它们应该随着时间不断增值。"

### 答辩亮点提炼

1. **从静态到动态**：书签不再是快照，而是持续演变的活数据
2. **分层数据源覆盖**：从结构化（GitHub API）到半结构化（B站/知乎）到非结构化（通用网页），展示工程广度
3. **智能调度**：自适应频率 + 优先级队列 + 速率限制，展示系统设计能力
4. **AI 预测**：简单线性回归但展示效果好，给用户"智能感"
5. **三层通知递进**：轻量 Dashboard → 详细动态页 → 重要推送，展示 UX 设计思考
6. **数据可视化**：Sparkline + Recharts 完整图表，展示前端技术广度

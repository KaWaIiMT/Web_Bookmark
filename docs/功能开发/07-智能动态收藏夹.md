# 功能：智能动态收藏夹

> 状态：规划完成 / 待开发
> 创建日期：2026-06-06
> 关联功能：[[AI自动分类]] [[关联推荐+学习路径]] [[侧边栏]]

---

## 一、概述

将 MarkBox 的静态收藏夹升级为「智能动态收藏夹」——基于规则引擎自动聚合符合条件的书签。用户设置规则（标签、分类、状态、时间等），符合条件的书签自动流入/流出。类似 Gmail Filter 而非传统文件夹：书签不是"被放进去"的，而是"符合条件自动归位"的。

---

## 二、技术选型

### 决策 1：规则引擎设计

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 预设条件组合器 — 提供固定字段的条件行（标签=、分类包含、状态=...），用户用 AND/OR 组合 | 实现简单直观，类似 Gmail Filter，用户学习成本低 | 表达能力有限，复杂规则（如"最近 30 天收藏的 Rust 或 Go 相关文章"）需要多行条件 |
| **B** | A + AI 自然语言规则 — 在 A 基础上，用户可以输入自然语言描述规则，AI 解析为条件组合 | 降低非技术用户门槛——"把所有 Rust 相关的待读视频收集起来" | AI 解析可能有歧义，需要预览确认机制 |
| **C** | A + B + 排序/数量限制 — 额外支持每个收藏夹的排序规则（按时间/热度/阅读状态）和数量上限 | 更完整的自动化——不仅是"收集哪些"，还有"怎么排列"和"最多多少" | 规则编辑 UI 复杂度进一步增加 |

> ✅ **推荐 C**。预设条件覆盖 80% 场景，AI 自然语言降低新用户门槛，排序和数量限制让自动化更完整（"只保留最近 100 篇 Rust 文章，按收藏时间倒序"）。排序规则复用现有书签排序逻辑，数量上限是简单的 LIMIT 查询。

> ❌ **排除 A**：缺少 AI 自然语言规则，纯条件组合器的表达力有限，且对新用户不够友好。
>
> ❌ **排除 B**：缺少排序和数量限制，收藏夹可能堆积过多书签，失去"精选"的意义。

---

### 决策 2：规则更新时机

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 仅用户手动点击「刷新」时重新匹配 | 无后台计算开销，用户完全掌控 | 收藏夹内容滞后，新增书签不会自动归类 |
| **B** | 书签创建/更新时触发匹配 + 定期全量重算（每日） | 新书签实时归位，定期重算处理规则变更后的存量书签 | 书签创建时增加一次规则匹配的数据库查询 |
| **C** | 纯实时 — 每次访问收藏夹时实时计算匹配结果 | 永远最新，无定时任务 | 收藏夹多的用户每次加载都要跑 N 条 SQL，性能风险 |

> ✅ **推荐 B**。书签创建/更新时触发增量匹配（单条书签对全部智能收藏夹做一次匹配，毫秒级），保证实时性。每日全量重算处理规则变更后的存量对齐。这是 Gmail Filter 的经典模式。

> ❌ **排除 A**：收藏夹成为"过期视图"，违背了智能自动化的核心价值。
>
> ❌ **排除 C**：当用户有 10+ 智能收藏夹时，每次加载侧边栏都要跑 10+ 条复杂查询，性能不可控。

---

### 决策 3：与现有手动收藏夹的共存策略

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 侧边栏独立区块「智能收藏夹」，与手动收藏夹并存，视觉区分（⚡规则图标 vs 📁文件夹图标） | 两种模式共存，用户按需使用。不破坏现有数据和使用习惯 | 侧边栏内容增加，需要合理的折叠/展开设计 |
| **B** | 替换现有收藏夹 — 所有收藏夹都升级为规则驱动（手动收藏夹=规则"手动添加的书签"） | 概念统一，用户只有一种心智模型 | 破坏性变更，现有手动收藏夹数据需要迁移 |
| **C** | 独立页面「规则中心」，与侧边栏收藏夹完全分离 | 功能隔离清晰，不影响现有侧边栏 | 两种收藏夹分散在两个地方，用户困惑"我该用哪个？" |

> ✅ **推荐 A**。侧边栏两个独立区块，图标区分（⚡ vs 📁），可折叠。用户已有的手动收藏夹不受影响。智能收藏夹区块默认显示匹配数量（如"Rust 待读 · 12"），点击展开查看列表。两种模式互补而非替代。

> ❌ **排除 B**：破坏性变更风险大，现有用户的手动收藏夹数据迁移可能丢失手动排序等细节信息。
>
> ❌ **排除 C**：功能分散导致用户困惑，且与侧边栏快速筛选的核心使用场景割裂。

---

### 决策 4：规则共享与模板（MVP 范围决策）

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 仅个人使用 — 规则仅创作者可见和使用 | 实现简单，无社交依赖 | 无法复用他人的优质规则 |
| **B** | 规则模板市场 — 用户可以发布规则模板，其他人一键导入 | 社区驱动，降低配置门槛 | 需要审核机制防止恶意规则，开发量显著增加 |

> ✅ **推荐 A**。MVP 阶段仅支持个人规则。规则配置本身已经足够简单（或通过 AI 自然语言生成），模板市场的价值在用户量增长后才显现。可在后续版本中评估。

> ❌ **排除 B**：模板市场需要审核、评分、报告机制，对于个人书签管理器的 MVP 阶段属于过度设计。

## 三、交互设计

### 3.1 规则配置流程

```
创建智能收藏夹
  → 命名（如"Rust 待读"）
  → 选择规则模式：
    - 可视化条件组合器（字段 + 运算符 + 值 + AND/OR）
    - AI 自然语言输入
  → 实时预览匹配结果（显示数量 + 前 3 条书签）
  → 设置排序规则（最新 / 最热 / 最久未读）
  → 设置数量上限（可选）
  → 保存 → 侧边栏出现新收藏夹
```

### 3.2 预设条件字段

| 字段 | 运算符 | 示例值 |
|---|---|---|
| 标签 | 包含任意/包含全部/不包含 | Rust, 异步 |
| 分类 | 属于/属于子类 | 技术学习 > Rust |
| 状态 | 等于/不等于 | 待读 |
| 内容类型 | 等于/不等于 | 视频, 文章 |
| 收藏时间 | 最近 N 天/N 天前 | 最近 30 天 |
| 域名 | 包含/等于 | github.com |
| 关键词 | 标题包含/摘要包含 | "tutorial" |

### 3.3 AI 自然语言规则

输入："把所有 Rust 相关的未读视频收集起来，按发布时间排序"

AI 解析预览：
```
✅ 已识别规则：
  · 标签包含：Rust
  · 状态：待读
  · 内容类型：视频
  · 排序：收藏时间倒序
📊 预计匹配：8 条书签
```

## 四、技术实现

### 4.1 数据模型

```prisma
model Collection {
  // ... 现有字段
  isSmart     Boolean   @default(false)
  rules       String?   // JSON: SmartCollectionRules | null
  sortBy      String?   // createdAt | updatedAt | title
  sortOrder   String?   // asc | desc
  maxItems    Int?      // 数量上限
}
```

```typescript
interface SmartCollectionRules {
  combinator: 'and' | 'or';
  conditions: RuleCondition[];
  aiPrompt?: string; // 记录原始 AI 自然语言输入
}

interface RuleCondition {
  field: 'tag' | 'category' | 'status' | 'contentType' | 'createdAt' | 'domain' | 'keyword';
  operator: 'contains' | 'not_contains' | 'equals' | 'not_equals' | 'within_days' | 'before_days';
  value: string | string[];
}
```

### 4.2 规则→Prisma Where 转换

```typescript
function rulesToPrismaWhere(rules: SmartCollectionRules): Prisma.BookmarkWhereInput {
  const clause = rules.conditions.map(c => conditionToPrisma(c));
  return rules.combinator === 'and' ? { AND: clause } : { OR: clause };
}
```

### 4.3 API 设计

| 端点 | 方法 | 说明 |
|---|---|---|
| `/api/collections` | POST | 创建（新增 isSmart/rules 参数） |
| `/api/collections/[id]` | PATCH | 更新规则 |
| `/api/collections/[id]/preview` | GET | 预览规则匹配结果 |
| `/api/collections/[id]/refresh` | POST | 手动刷新智能收藏夹 |

## 六、答辩亮点

1. **从文件夹到规则引擎**：概念升级——"不是放在哪里，而是符合什么条件"
2. **Gmail Filter 模式**：业界验证的产品范式，有成熟的对标分析
3. **AI 自然语言规则**：降低非技术用户门槛，展示 AI 在产品细节中的应用
4. **增量+全量双模式更新**：展示对数据一致性工程问题的思考

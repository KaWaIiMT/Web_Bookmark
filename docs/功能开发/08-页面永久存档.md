# 功能：页面永久存档

> 状态：规划完成 / 待开发
> 创建日期：2026-06-06
> 关联功能：[[AI自动分类]] [[链接健康监控]] [[书签详情]] [[AI知识问答(RAG)]]

---

## 一、概述

为 MarkBox 新增「页面永久存档」功能——收藏书签时异步抓取并存储一份简化 HTML 快照，防止原始页面被删除、修改或失效导致知识丢失。用户在书签详情中可在"原始页面预览"和"存档快照"之间一键切换。

这是对标 Raindrop.io Permanent Copy 的标杆功能。与已有的元数据提取引擎（cheerio）共享抓取逻辑，与 Feature E（链接健康监控）形成闭环。

---

## 二、技术选型

### 决策 1：存档内容深度

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 仅纯文本 — 抓取后提取正文纯文本存储 | 体积最小（5-20KB/条），搜索和 RAG 友好，无外部依赖 | 丢失排版、图片、代码块格式，可读性差 |
| **B** | 简化 HTML — 去 script/style/iframe/事件处理器，保留正文 HTML 结构 + 图片引用 | 可渲染查看，保留排版和图片，体积可控（50-200KB/条）。与已有 cheerio 抓取逻辑复用 | 外部图片链接可能仍失效（源站删除图片） |
| **C** | 完整截图 + 纯文本 — 用 Puppeteer 截取全页 PNG + 提取纯文本 | 视觉保真 100%，图片链接全部失效也不怕 | 截图体积大（500KB-2MB），需要 Chromium 运行时，部署复杂度高 |

> ✅ **推荐 B**。简化 HTML 保留页面结构和可读性，是性价比最高的方案。与已有的 cheerio 元数据抓取逻辑高度复用。图片失效问题可通过"存档时同步下载封面图到本地"部分缓解。当项目规模增长后可升级为 C（截图兜底）。

> ❌ **排除 A**：纯文本丢失了代码块格式、表格、排版等关键信息，对于技术文章（代码块）和结构化内容（表格）可读性极差。
>
> ❌ **排除 C**：Puppeteer 需要额外的 Chromium 运行时（~300MB），在个人书签管理器的部署环境中性价比不足。且全页截图对长页面（如长文博客）可能截断。

---

### 决策 2：存档触发时机

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 书签创建时同步抓取 | 确保每篇书签创建完成时已有存档，零遗漏 | 增加收藏等待时间（+2-5s 网络抓取），用户体验下降 |
| **B** | 书签创建后异步后台抓取 | 不阻塞收藏流程，收藏体验不受影响 | 部分书签可能因网络问题抓取失败，用户需被告知 |
| **C** | 仅用户手动触发「存档」按钮 | 用户按需存档，无后台资源浪费 | 依赖用户操作习惯，大量书签可能被遗忘存档 |

> ✅ **推荐 B**。异步抓取保持收藏流程的流畅体验（<1s 完成收藏），与 AI 分类/摘要的异步模式一致。配合 Feature E（链接健康监控）可检测抓取失败的书签。书签详情中显示存档状态标识（✅ 已存档 / ⏳ 存档中 / ❌ 存档失败 / — 未存档）。

> ❌ **排除 A**：同步抓取将收藏时间从 <1s 延长到 3-6s，严重损害核心体验——"一键收藏"的流畅感是 MarkBox 的核心竞争力。
>
> ❌ **排除 C**：纯手动触发会导致大量书签处于"无存档"状态，存档覆盖率极低，失去功能价值。

---

### 决策 3：存档渲染安全性

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 仅展示纯文本，不渲染 HTML | 绝对安全，零 XSS 风险 | 丢弃了 HTML 存档的排版和可读性优势 |
| **B** | `<iframe sandbox>` 隔离渲染存档 HTML | 浏览器原生安全边界，零信任渲染。sandbox 阻止脚本执行、表单提交、弹窗 | 部分 CSS（如 `position: fixed`）在 sandbox 中可能表现异常 |
| **C** | 服务端 DOMPurify 净化 + 直接渲染到页面 DOM | 完全控制渲染样式，无 iframe 限制 | 净化不彻底时有 XSS 风险，安全责任在开发者而非浏览器 |

> ✅ **推荐 B**。`<iframe sandbox="allow-same-origin">` 是浏览器级别的安全隔离——即使存档 HTML 中包含恶意脚本也无法执行。比 C 更安全（安全由浏览器保证），比 A 更实用（保留排版）。CSS 异常是极小概率的边界 case，可通过净化时移除 `position: fixed/sticky` 来解决。

> ❌ **排除 A**：丢弃了 HTML 存档的核心价值——可渲染查看。用户看到纯文本的代码块（无语法高亮）和破碎的表格，体验等同于"没存档"。
>
> ❌ **排除 C**：DOMPurify 是优秀的工具但无法保证 100% 安全（零日漏洞、配置遗漏）。安全是存档功能的基础要求，不值得冒险。

---

### 决策 4：存储位置

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 数据库 TEXT 字段（libsql） | 零运维，与现有架构一致。部署简单 | 大规模（>10k 条）时数据库体积增长，备份时间延长 |
| **B** | 文件系统存储（本地 `.data/archives/` 目录） | 数据库保持轻量，文件系统 I/O 效率高 | 增加运维复杂度（备份需要同时备份 DB 和文件），文件路径管理 |
| **C** | 对象存储（S3/MinIO/Cloudflare R2） | 无限扩展，CDN 加速 | 架构复杂度显著增加，需要额外服务。个人项目不匹配 |

> ✅ **推荐 A**。个人书签管理器规模（<10k 条）下，简化 HTML（<200KB/条）+ 纯文本总计 <2GB，SQLite 完全可以承受。零运维是个人项目的关键优势。后续如果数据量增长（>50k 条），可迁移到 B 或 C。

> ❌ **排除 B**：增加运维复杂度（备份策略需要覆盖两个存储层），对于 MVP 阶段不值得。
>
> ❌ **排除 C**：对象存储是典型的过度架构。为个人书签管理器引入 S3/MinIO 违背了项目的简洁部署原则。

---

### 决策 5：存档更新策略

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 一次存档，永不更新 | 实现最简单，零额外抓取 | 存档内容随时间失真——页面可能更新、修改、重定向 |
| **B** | 手动触发更新 + 自动刷新高频书签（30 天内收藏） | 低频内容不浪费资源，高频内容保持新鲜 | 30 天阈值可能不够精确 |
| **C** | 全量定期刷新（如每月重抓全部书签存档） | 所有存档保持最新 | 资源浪费——大量页面不会变化（如 GitHub 仓库首页、已归档文章） |

> ✅ **推荐 B**。手动触发给用户主动权（"这篇文章更新了，刷新存档"）。自动刷新仅针对最近 30 天收藏的书签（这些内容更可能仍在更新），频率每周一次。30 天后降级为每季度一次或手动触发。兼顾新鲜度和资源效率。

> ❌ **排除 A**：存档会逐渐失真，特别是技术文档类页面（如官方文档更新 API 用法）。
>
> ❌ **排除 C**：全量刷新对大量未变化页面进行无效抓取，浪费带宽和计算资源，也增加了被目标网站限流的风险。

## 三、交互设计

### 3.1 书签详情中的存档切换

BookmarkDetailSheet 中新增 Tab 切换：

```
[📋 预览] [📦 存档快照] [🔗 相关推荐]
```

- **预览 Tab**：现有封面图 + AI 摘要（不变）
- **存档快照 Tab**：在 `<iframe sandbox>` 中渲染存档 HTML，顶部显示存档时间
- 原始链接 404 → 自动切换到存档 Tab + 顶部提示"原始页面已失效，以下为最近存档"
- 从未存档 → 显示「创建存档」按钮 + 说明文案

### 3.2 存档状态标识

- ✅ 已存档（绿色）：显示存档时间
- ⏳ 存档中（黄色脉冲）：显示进度动画
- ❌ 存档失败（红色）：显示失败原因 + 「重试」按钮
- — 未存档（灰色）：显示「创建存档」按钮（仅手动触发模式下）

### 3.3 设置页存档管理

- 已存档数 / 总书签数
- 存储用量（如 "总计 45.2 MB"）
- 批量操作：刷新最近 30 天存档
- 按时间筛选待更新存档

## 四、技术实现

### 4.1 数据模型

```prisma
model Bookmark {
  // ... 现有字段
  archiveHtml   String?   // 简化版 HTML 快照
  archiveText   String?   // 提取的纯文本（用于搜索/RAG）
  archivedAt    DateTime? // 存档时间
  archiveStatus String?   // pending | success | failed | null
}
```

### 4.2 存档抓取

```typescript
// src/lib/archiver.ts
async function archivePage(url: string): Promise<{ html: string; text: string }> {
  const res = await fetch(url, {
    headers: { 'User-Agent': 'MarkBox-Archiver/1.0' },
    signal: AbortSignal.timeout(15000),
  });
  const rawHtml = await res.text();
  const $ = cheerio.load(rawHtml);

  // 移除危险标签
  $('script, style, iframe, noscript, [onclick], [onerror], [onload]').remove();

  // 提取正文区域
  const main = $('article, main, [role="main"], .content, .post-body, .markdown-body');
  const html = (main.length ? main.html() : $('body').html()) || '';
  const text = (main.length ? main.text() : $('body').text())
    ?.replace(/\s+/g, ' ').trim().slice(0, 50000) || '';

  return { html, text };
}
```

### 4.3 API 设计

| 端点 | 方法 | 说明 |
|---|---|---|
| `/api/bookmarks/[id]/archive` | POST | 触发存档（异步） |
| `/api/bookmarks/[id]/archive` | GET | 获取存档内容 |
| `/api/bookmarks/[id]/archive/status` | GET | 获取存档状态 |

## 六、答辩亮点

1. **知识永久性**：对标 Raindrop.io Permanent Copy，展示对"链接失效"问题的系统性解决方案
2. **安全渲染**：`<iframe sandbox>` 浏览器级安全隔离，展示对 Web 安全的理解
3. **异步架构**：不阻塞核心收藏体验，与 AI 分类/摘要共享异步管道设计
4. **与链接健康监控联动**：存档 + 失效检测形成完整的"知识保鲜"体系

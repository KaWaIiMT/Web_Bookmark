# 功能：同一主题横向对比

> 状态：规划完成 / 待开发
> 创建日期：2026-06-05
> 关联功能：[[AI自动分类]] [[关联推荐+学习路径]] [[知识图谱视图]] [[书签分享]]

---

## 一、概述

为 MarkBox 新增「横向对比」功能。当用户收藏了同一主题的多篇文章时，AI 自动检测并提示发起对比分析。从核心观点、技术方案、元信息等多个维度进行系统对比，结果以雷达图 + 对比矩阵表 + AI 综合评语的形式呈现。支持保存对比记录、导出 Markdown/图片报告、分享给协作者。

这个功能是答辩的核心亮点之一——一张对比矩阵表直观展示 AI 的理解和分析能力，比单纯的"收藏+分类"高一个维度。

---

## 二、技术选型

| 决策点 | 选择 | 理由 |
|---|---|---|
| **对比触发方式** | C：AI 自动检测 + 手动选择混合 | AI 自动发现降低使用门槛（"你可能想对比这些内容"），手动多选给予灵活度。两者进入同一个对比分析流程 |
| **对比维度** | C：核心观点 + 技术方案 + 元信息全维度 | 观点矩阵回答"谁说了什么"，技术细节回答"怎么做"，元信息回答"谁更权威、谁更新"。覆盖从浏览到决策的完整链路 |
| **可视化呈现** | C：雷达图 + 对比矩阵表 + AI 综合评语 | 雷达图一图看懂差异（答辩开场），矩阵表深入细节（答辩展开），AI 评语给出 actionable 建议（答辩收尾）。层次递进 |
| **保存与复用** | C：持久化保存 + Markdown/图片导出 + 分享链接 | 对比不是一次性消费品，而是知识资产。复用现有 share token 机制，生成可分享的对比报告链接 |
| **AI 模型** | DeepSeek V4（复用现有 `deepseek.ts` 客户端） | 对比分析需要深度理解和结构化输出，DeepSeek V4 的能力足够。使用 JSON Mode 或 Structured Output 保证返回格式稳定 |
| **雷达图渲染** | Recharts 3.8（已在项目依赖中） | 项目已使用 Recharts 做 Dashboard 图表，雷达图是其内置图表类型。保持技术栈一致性 |

### 模块 A：对比触发

#### A.1 AI 自动检测（被动发现）

系统在后台定期（或在新书签创建时）检测可对比的内容集群：

**检测逻辑：**

```
扫描用户书签库
  → 按标签共现 + 语义相似度聚类
  → 发现满足条件的集群：
    ✅ 至少 2 篇书签共享 ≥ 2 个标签
    ✅ 语义相似度 ≥ 0.75
    ✅ 非同一 URL（排除同一篇文章的不同收藏）
  → 对每个集群，检查是否已有对比记录（避免重复提示）
  → 在 Dashboard 或书签列表顶部展示提示卡片
```

**提示卡片设计：**

```
┌────────────────────────────────────────────┐
│  🔍 发现可对比内容                          │
│                                            │
│  你收藏了 6 篇关于「React Server            │
│  Components」的文章，它们来自不同作者，      │
│  观点可能不同。是否查看对比分析？            │
│                                            │
│  涉及书签：                                 │
│  · RSC 实战指南 — 2024-03                  │
│  · RSC 深度解析 — 2025-01                  │
│  · 为什么我不推荐 RSC — 2024-08            │
│  ... 共 6 篇                               │
│                                            │
│  [查看对比分析]  [忽略]  [稍后提醒]          │
└────────────────────────────────────────────┘
```

**触发时机：**
- 新书签创建后，异步检查是否形成新的可对比集群（不影响收藏流程）
- 用户进入 Dashboard 时，检查是否有未读的对比提示
- 每周报告中加入"本周可对比主题"板块

#### A.2 手动选择发起（主动操作）

用户在以下场景可以手动多选书签并发起对比：

- **书签列表（Grid/Gallery 视图）**：长按或勾选进入多选模式，选中 2-5 篇 → 工具栏出现「对比分析」按钮
- **知识图谱视图**：框选多个书签节点（Shift + 拖动）→ 右键菜单 → 「对比选中书签」
- **书签详情页**：在详情页底部「相关推荐」区域，提供「与当前书签对比」快捷按钮
- **搜索页面**：搜索结果中勾选多篇 → 「对比分析」

**数量限制：**
- 最少 2 篇，最多 5 篇（对比维度多，超过 5 篇矩阵可读性下降）
- 超过 5 篇时提示"请选择 2-5 篇书签进行对比"

### 模块 B：对比分析维度

#### B.1 核心观点层

| 子维度 | 说明 | 示例 |
|---|---|---|
| **核心论点** | 文章的 main argument / thesis | "RSC 是 React 的未来" vs "RSC 不适合所有场景" |
| **技术立场** | 支持/反对/中立 + 立场强度 | 强烈支持（⭐⭐⭐⭐⭐）/ 谨慎中立（⭐⭐⭐）/ 明确反对（⭐⭐） |
| **关键论据** | 支撑核心论点的 2-3 个关键论据 | 论据 1：性能提升 30%；论据 2：开发体验改善 |
| **适用场景** | 作者认为该方案适用于什么场景 | "适合内容型网站" vs "适合所有 React 应用" |

#### B.2 技术方案层

| 子维度 | 说明 | 示例 |
|---|---|---|
| **技术方案描述** | 文章提出或讨论的技术方案 | 方案 A：全量 SSR + Streaming；方案 B：CSR + Selective Hydration |
| **代码示例对比** | 不同文章对同一问题的代码解法差异 | 文章 A 用 `use server` 指令；文章 B 用传统 API Route |
| **性能数据** | 文中引用的 benchmark/性能数据 | 文章 A 声称 TTFB 降低 40%；文章 B 实测 TTFB 降低 15% |
| **依赖/生态** | 方案所依赖的技术栈或框架版本 | 需要 Next.js 14+；兼容 React 18 |

#### B.3 元信息层

| 子维度 | 说明 | 示例 |
|---|---|---|
| **作者/来源权威性** | AI 评估来源的权威程度 | 官方文档（⭐⭐⭐⭐⭐）/ 知名技术博客（⭐⭐⭐⭐）/ 个人博客（⭐⭐⭐） |
| **发布时间** | 时效性分析 | 2024-03 vs 2025-01 —— 后者更新 |
| **内容深度** | AI 评估内容的深度等级 | 入门概览 / 实战指南 / 深度原理分析 |
| **可读性** | AI 评估内容的阅读难度 | 适合新手 / 需要一定基础 / 专家级 |
| **内容类型** | 视频 / 文章 / 论文 / 官方文档 | 不同介质的信息密度和可信度不同 |
| **覆盖完整度** | 文章对该主题的覆盖范围 | 仅介绍基础 / 覆盖核心概念 / 全面深入 |

#### B.4 AI 综合评语

基于以上维度，AI 生成一段综合评语：

```
综合评语：

这 6 篇文章代表了 React Server Components 社区的三种主流观点：
1. 积极派（A、D、F）：认为 RSC 是未来趋势，建议立即采用
2. 谨慎派（B、E）：承认优势但指出迁移成本和不适用场景
3. 反对派（C）：列举了 RSC 的不足之处，推荐等待生态成熟

建议阅读顺序：B（了解全貌）→ A（了解优势）→ C（了解缺陷）→ D（深入实战）

综合推荐入门文章：B（2025-01，深度适中，权威性高，观点中立）
```

#### C.1 顶部：雷达图概览

在对比页面顶部，使用 Recharts 雷达图（RadarChart）展示多维度评分对比：

**雷达图维度轴：**
- 深度（Depth）
- 可读性（Readability）
- 权威性（Authority）
- 时效性（Freshness）
- 代码质量/实用性（Practicality）

每篇文章用不同颜色的多边形表示。一眼看出各文章的侧重点差异：
- "文章 A 的深度和权威性很高，但可读性差"

- 悬停某个顶点 → tooltip 显示具体评分和说明
- 点击图例中的某篇文章 → 高亮/隐藏该多边形
- 点击某个维度轴 → 下方矩阵表滚动到对应行

### 4.1 数据模型

```prisma
// 对比记录
model Comparison {
  id          String   @id @default(cuid())
  userId      String
  title       String                    // AI 生成的对比标题
  bookmarkIds String                    // JSON: ["id1", "id2", "id3"]

  shareToken  String?  @unique         // 分享 token
  shareExpiry String?                   // 分享过期时间
  createdAt   String   @default(now())
  
  user        User     @relation(fields: [userId], references: [id], onDelete: Cascade)
}
```

**对比分析结果数据结构：**

```typescript
interface ComparisonResult {
  // 雷达图评分（每个维度 1-5 分）
  radarScores: {
    bookmarkId: string;
    scores: {
      depth: number;           // 深度
      readability: number;     // 可读性
      authority: number;       // 权威性
      freshness: number;       // 时效性
      practicality: number;    // 实用性
    };
  }[];
  
  // 对比矩阵
  matrix: ComparisonMatrix;
  
  // AI 综合评语
  aiCommentary: {
    opinionDistribution: string;   // 观点分布描述
    keyDivergences: string[];      // 关键分歧点
    recommendedOrder: string[];    // 建议阅读顺序 (bookmarkIds)
    conclusion: string;            // 推荐结论
  };
}

interface ComparisonMatrix {
  dimensions: ComparisonDimension[];
  columns: { bookmarkId: string; title: string }[];
  cells: {
    dimensionId: string;
    bookmarkId: string;
    value: string;              // 单元格内容
    significance: 'normal' | 'notable' | 'critical'; // 差异显著度
  }[];
}
```

### 4.3 核心算法

#### 可对比集群检测

```typescript
// src/lib/comparisons/detect-clusters.ts

interface ClusterCandidate {
  topicLabel: string;          // 主题名称（如 "React Server Components"）
  bookmarkIds: string[];       // 候选书签 ID 列表
  sharedTagCount: number;      // 共享标签数
  avgSimilarity: number;       // 平均语义相似度
  hasDivergence: boolean;      // 是否有观点分歧（立场不同才值得对比）
}

async function detectComparableClusters(
  userId: string
): Promise<ClusterCandidate[]> {
  // 1. 找出用户所有标签中，关联书签 ≥ 2 的标签组
  const tagGroups = await prisma.tag.findMany({
    where: { userId },
    include: {
      bookmarks: {
        include: { bookmark: { select: { id: true, embedding: true } } },
      },
    },
  });
  
  const clusters: ClusterCandidate[] = [];
  
  for (const tag of tagGroups) {
    if (tag.bookmarks.length < 2) continue;
    
    const bookmarkIds = tag.bookmarks.map(b => b.bookmarkId);
    
    // 2. 计算组内平均语义相似度
    const similarities = await calculatePairwiseSimilarities(
      tag.bookmarks.map(b => ({ 
        id: b.bookmark.id, 
        embedding: b.bookmark.embedding 
      }))
    );
    const avgSimilarity = average(similarities);
    
    // 3. 过滤：相似度 ≥ 0.75
    if (avgSimilarity < 0.75) continue;
    
    // 4. 检查是否已有对比记录
    const existingComparison = await prisma.comparison.findFirst({
      where: {
        userId,
        bookmarkIds: { contains: bookmarkIds[0] }, // 简化检查
      },
    });
    if (existingComparison) continue;
    
    clusters.push({
      topicLabel: tag.name,
      bookmarkIds,
      sharedTagCount: tag.bookmarks.length,
      avgSimilarity,
      hasDivergence: true, // 需要 AI 进一步判断
    });
  }
  
  // 5. 按共享标签数 + 相似度排序，返回 Top-5
  return clusters
    .sort((a, b) => 
      (b.sharedTagCount * 0.6 + b.avgSimilarity * 0.4) -
      (a.sharedTagCount * 0.6 + a.avgSimilarity * 0.4)
    )
    .slice(0, 5);
}
```

async function analyzeComparison(

function buildComparisonPrompt(bookmarks: BookmarkData[]): string {
  const articles = bookmarks.map((b, i) => {

  }).join('\n---\n');

## 五、UI/UX 设计要点

### 5.1 雷达图设计

- **配色**：每篇文章使用 ESG 调色板中的不同颜色，Alpha 0.2 填充 + 1.0 描边
- **动画**：雷达图多边形从中心向外展开（spring 动画，~800ms）
- **网格线**：半透明同心五边形（1-5 刻度）
- **标签**：维度标签使用中文，字号略小于正文
- **悬停效果**：鼠标悬停某个多边形时轻微放大（scale 1.05）+ 其他多边形降低透明度

### 5.2 矩阵表设计

- **表头固定**：Sticky header，滚动时始终可见
- **首列固定**：维度名列 Sticky left，水平滚动时不离开视图
- **差异高亮**：
  - 立场相反的单元格：红色边框 + 浅红背景（`rgba(239, 68, 68, 0.08)`）
  - 差异显著的单元格：黄色边框 + 浅黄背景
  - 一致的单元格：绿色小对勾图标
- **分组**：三个维度组（观点层/技术层/元信息层）用不同底色区分，组间有分隔行
- **响应式**：小屏设备（<1024px）改为纵向堆叠卡片，每篇文章一张卡片，差异用箭头标注

### 5.3 加载态

- AI 分析期间显示对比分析进度动画：
  - 阶段 1：「正在提取文章关键信息...」（1s）
  - 阶段 2：「正在对比观点差异...」（1-2s）
  - 阶段 3：「正在生成综合评语...」（1s）
- 每个阶段有对应的 Lottie/动画图标
- 如果 AI 分析超过 10s → 显示"分析耗时较长，请耐心等待..."+ 趣味加载文案

### 5.4 空态与边缘情况

- **相似度太高（观点无分歧）**：AI 提示"这几篇文章观点高度一致，建议选择立场更丰富的文章进行对比"。仍然展示对比矩阵，但特殊标注"高度一致"。
- **标签相同但内容完全不同**：AI 检测到书签虽然共享标签但实际讨论不同子主题 → 提示"这些书签似乎讨论的是'{标签}'的不同方面，对比价值可能有限"
- **部分书签 AI 摘要缺失**：降级到仅使用标题 + 描述进行对比，结果标注"部分书签信息不完整，对比可能不够深入"

## 七、答辩演示脚本

**场景：展示 AI 的深度理解能力**

> "我们不仅能用 AI 分析和分类书签，还能让它做更高级的事情——横向对比。
>
> 这里我收藏了 6 篇关于 React Server Components 的文章，来自不同作者、不同时间。AI 自动检测到这个可对比的集群，并在 Dashboard 上提示了我。
>
> 点击进入对比页面。首先看到雷达图——一图看懂 6 篇文章的侧重点差异。A 文章深度高但可读性低，B 文章各方面均衡，C 文章权威性不足但观点独特。
>
> 往下是详细的对比矩阵。AI 从核心观点、技术方案、元信息三个维度进行了系统对比。注意——这里红色的单元格表示这些文章在这个维度上存在显著分歧。比如对于"RSC 是否适合所有项目"，A 和 C 的观点完全相反。
>
> 底部是 AI 的综合评语，给出了观点分布、关键分歧点、建议阅读顺序和最终推荐。
>

### 答辩亮点提炼

1. **多维度结构化对比**：不只是"这两篇相似"，而是从观点、技术、元信息三个层次深度分析
2. **可视化层次递进**：雷达图（概览）→ 矩阵表（细节）→ AI 评语（洞察），叙事节奏完整
3. **差异自动高亮**：AI 自动识别分歧点并在矩阵中标注，不是死板的表格
4. **知识资产化**：对比结果可以保存、导出、分享，不只是消费级功能
5. **自动发现**：AI 主动检测可对比内容，降低使用门槛——"你可能想对比这些内容"

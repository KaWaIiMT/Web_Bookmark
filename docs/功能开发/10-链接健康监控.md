# 功能：链接健康监控

> 状态：规划完成 / 待开发
> 创建日期：2026-06-06
> 关联功能：[[页面永久存档]] [[仪表盘]] [[热度追踪]] [[书签详情]]

---

## 一、概述

为 MarkBox 新增「链接健康监控」功能——自动检测所有活跃书签的链接状态（HTTP 状态码、重定向、内容变化），在书签卡片、Dashboard 和邮件通知三层递进展示。检测到失效链接时自动关联页面存档（Feature C），让用户知道哪些知识资产仍然可访问、哪些需要关注。

这是对标 Raindrop.io Broken Links 的标杆功能。与页面存档功能形成"检测 → 保护"的完整知识保鲜闭环。

---

## 二、技术选型

### 决策 1：监控范围

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 全量监控 — 检查用户所有书签的链接状态 | 无死角，每条链接都受保护 | 大量归档或低频书签浪费资源 |
| **B** | 按优先级分级 — 活跃书签（近 30 天收藏 + 未读/在读状态）高频检查，归档书签低频或手动触发 | 资源效率高，聚焦用户最关心的内容 | 归档书签也需要偶尔检查——用户可能某天需要回顾 |
| **C** | 全量 + 分级频率 — 活跃每天、普通每周、归档每月 | 最全面，兼顾效率和覆盖面 | 调度逻辑更复杂，MVP 阶段过度设计 |

> ✅ **推荐 B**。MVP 阶段优先监控活跃书签（近期收藏 + 未读/在读），这是用户最关心的知识资产。归档书签可在设置页手动触发检查。后续可根据用户反馈升级为全量分级监控。

> ❌ **排除 A**：全量检查大量归档书签是对计算和网络资源的浪费——用户可能几年都不会访问它们。
>
> ❌ **排除 C**：全量+分级是理想方案，但 MVP 阶段应先验证核心场景（活跃书签的监控价值），再扩展范围。

---

### 决策 2：检测深度

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 仅 HTTP 状态码 — HEAD 请求检查返回状态（200/301/302/404/500/超时） | 最轻量，一次 HEAD 请求即可，对目标服务器负担最小 | 不检测内容变化——页面可能被替换、被停放页面接管但仍返回 200 |
| **B** | A + 内容变化检测 — 定期 GET 请求检查标题/描述是否变化 + 重定向目标域名是否变更 | 更全面的健康评估。重定向检测特别重要——被劫持到停放站的 URL 返回 200 但域名变了 | GET 请求流量更大，但可通过降低检测频率（如每 10 次 HEAD 做一次 GET）来控制 |
| **C** | B + SSL 证书过期 + 域名注册过期检测 | 安全维度最全面 | 偏向运维工具，与个人书签管理的核心场景不匹配 |

> ✅ **推荐 B**。HEAD 请求做日常快速状态检测（占 90% 的检查），每 10 次检查做一次 GET 请求检测标题/描述是否与收录时一致。重定向检测是核心——URL 被劫持或域名过期后常被重定向到广告/停放页面，返回 200 但内容已被替换。与 Feature C（页面存档）联动，当检测到内容变化时提示用户"页面内容已更新，是否刷新存档？"。

> ❌ **排除 A**：仅 HTTP 状态码无法检测到"页面还在但内容已被替换"的情况——这是书签管理器最常见的链接失效形式之一（域名过期被停放）。
>
> ❌ **排除 C**：SSL 证书过期检测和域名注册过期检测偏向运维监控工具，与"我的书签还能不能看"的核心场景关联不大。过度设计。

---

### 决策 3：通知与呈现策略

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 仅 Dashboard 健康面板 — 在 Dashboard 显示链接健康概览（健康/重定向/失效数量） | 不打扰用户，主动查看时可见 | 用户可能长时间不检查 Dashboard，失效链接积压而不自知 |
| **B** | A + 书签卡片状态图标 — 每张书签卡片右上角显示健康状态（🟢 正常 / 🟡 重定向 / 🔴 失效） | 用户在浏览书签列表时即可感知健康状态，无需进入任何面板 | 所有卡片都需要渲染状态图标，对数据库查询有影响 |
| **C** | B + 邮件/推送通知（仅 404 完全失效） — 三层递进：卡片感知 → Dashboard 概览 → 紧急通知 | 最重要的事件（完全失效）能及时触达用户，且不过度打扰 | 邮件通知可能被用户忽视或标记为垃圾邮件 |

> ✅ **推荐 C**。三层递进覆盖从"低调感知"到"紧急触达"的完整链路：
> - **Layer 1（被动感知）**：书签卡片网角状态图标，浏览时自然看到
> - **Layer 2（主动查看）**：Dashboard 健康面板，概览统计 + 失效列表
> - **Layer 3（紧急触达）**：仅 404 完全失效时发送邮件/推送通知——这是知识资产的损失，需要用户行动

> ❌ **排除 A**：纯被动展示，用户可能数周不查看 Dashboard，失效链接持续堆积。
>
> ❌ **排除 B**：缺少 404 的紧急触达渠道——完全失效是知识损失，值得主动通知用户。邮件仅用于最高优先级（404），不会造成通知疲劳。

### 决策 4：与页面存档的联动

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 独立运作 — 健康监控和页面存档是两个独立功能，互不依赖 | 概念清晰，开发和测试独立 | 检测到失效时用户需手动确认是否有存档——"幸好存了"或"糟了没存"都应该是自动化的 |
| **B** | 失效时自动提示存档状态 — Dashboard 显示"失效链接 (3) · 有存档 (2) · 无存档 (1)"，点击失效书签弹出"原始链接已失效，是否查看存档？" | 自动关联，信息完整。用户无需在两个功能间手动切换 | 需要跨功能查询（健康监控检查 Bookmark.archiveHtml 是否存在） |
| **C** | B + 失效链接自动切换到存档视图（用户点击失效书签直接展示存档） | 最完整联动，用户无感知 | 改变书签详情默认行为，可能让用户困惑——"为什么点开看到的内容和上次不同？" |

> ✅ **推荐 B**。Dashboard 健康面板中直接展示"失效中已存档 vs 未存档"的细分统计，让用户一目了然。点击失效书签时弹出选择提示（"原始链接已失效，是否查看存档？" -> [查看存档] [尝试打开原始链接] [取消]），尊重用户选择权。

> ❌ **排除 A**：两个功能独立运作——检测到失效却不知道有没有存档、有存档却不知道链接是否失效——各自价值都打折。联动是这两个功能的乘数效应。
>
> ❌ **排除 C**：自动切换行为可能让用户困惑——"我点的是书签标题，为什么看到的是不同的内容？"。让用户知情并主动选择，比静默替换更符合 UX 原则。

### 决策 5：误报过滤策略

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 单次检测即报告 — 第一次检测到非 200 就立即标记并通知 | 响应最快，用户第一时间知道 | 网络抖动、目标站临时维护、CDN 波动导致大量误报——消耗用户信任 |
| **B** | 2 次确认（间隔 1 小时）— 首次检测到异常后等待 1 小时再次确认，两次都异常才标记 | 过滤绝大多数网络抖动误报（临时 DNS 故障、短暂 503），确认时间可接受 | 1 小时的确认延迟——但临时故障通常在几分钟内恢复 |
| **C** | 3 次确认指数退避（1h → 6h → 24h）— 三次都异常才最终确认 | 几乎完全消除误报 | 确认周期长达 31 小时，对于确凿的 404 来说确认太慢 |

> ✅ **推荐 B**。两次确认之间间隔 1 小时，可以过滤绝大多数由临时网络故障（DNS 抖动、目标站短暂维护重启、CDN 切换）导致的瞬态异常。1 小时的确认延迟是可接受的——真正的 404 不会在 1 小时内恢复。对于超时类异常（`ETIMEDOUT`），使用更宽松的策略（间隔 4 小时再次确认，可能只是目标站临时过载）。

> ❌ **排除 A**：网络抖动导致的假失效会持续消耗用户信任——"怎么又报失效，明明能打开"——几次误报后用户会完全忽略健康监控通知。
>
> ❌ **排除 C**：31 小时的确认周期对于真正的失效链接来说太长。用户希望在一天内知道链接是否真的 404 了。

## 三、交互设计

### 3.1 书签卡片健康状态

每张书签卡片右上角显示健康状态图标：

| 状态 | 图标 | 含义 |
|---|---|---|
| 🟢 正常 | 绿色小圆点 | HTTP 200，内容无变化 |
| 🟡 注意 | 黄色小圆点 | 301/302 重定向，或标题/描述有变化 |
| 🔴 失效 | 红色小圆点 | 404/500/超时（已二次确认） |
| ⬜ 未知 | 灰色小圆点 | 尚未检查或不在监控范围内 |

- 悬停图标 → tooltip 显示上次检查时间和状态详情
- 点击图标 → 弹出详情（HTTP 状态码 + 检查时间 + 如有存档则显示存档状态）

## 四、技术实现

### 4.1 数据模型

```prisma
model Bookmark {
  // ... 现有字段，新增：
  linkStatus        String?   // healthy | redirect | broken | unknown
  linkStatusCode    Int?      // 最近 HTTP 状态码
  linkCheckedAt     DateTime? // 最近检查时间
  linkCheckError    String?   // 最近错误信息（如 ETIMEDOUT）
  linkRedirectUrl   String?   // 重定向目标 URL
  linkTitleChanged  Boolean?  // 标题是否有变化
}
```

### 4.2 检测流程

```typescript
// src/lib/health-checker.ts

async function checkLinkHealth(bookmark: Bookmark): Promise<HealthResult> {
  // 1. HEAD 请求快速检测状态码
  let response: Response;
  try {
    response = await fetch(bookmark.url, {
      method: 'HEAD',
      redirect: 'manual',  // 不自动跟随重定向
      signal: AbortSignal.timeout(10000),
    });
  } catch (err) {
    return { status: 'broken', error: err.message };
  }

  // 2. 分类状态
  if (response.status === 200) {
    // 每 10 次 HEAD 做一次 GET 检测内容变化
    const shouldDeepCheck = Math.random() < 0.1;
    if (shouldDeepCheck) {
      return await deepContentCheck(bookmark);
    }
    return { status: 'healthy', code: 200 };
  }
  
  if ([301, 302, 307, 308].includes(response.status)) {
    const location = response.headers.get('location') || '';
    const newDomain = new URL(location).hostname;
    const oldDomain = new URL(bookmark.url).hostname;
    // 跨域名重定向 → 可疑
    if (newDomain !== oldDomain) {
      return { status: 'redirect', code: response.status, redirectUrl: location, suspicious: true };
    }
    return { status: 'redirect', code: response.status, redirectUrl: location };
  }
  
  if (response.status >= 400) {
    return { status: 'broken', code: response.status };
  }
  
  return { status: 'unknown', code: response.status };
}
```

### 4.3 误报过滤

```typescript
async function confirmBrokenLink(
  bookmarkId: string,
  firstCheck: HealthResult
): Promise<boolean> {
  if (firstCheck.status !== 'broken') return false;

  // 等待 1 小时
  await new Promise(resolve => setTimeout(resolve, 3600000));

  // 第二次确认
  const secondCheck = await checkLinkHealth(bookmark);
  
  // 两次都异常 → 确认失效
  return secondCheck.status === 'broken';
}
```

### 4.4 API 设计

| 端点 | 方法 | 说明 |
|---|---|---|
| `/api/health/overview` | GET | Dashboard 健康概览（统计 + 失效列表） |

## 六、答辩亮点

1. **对标 Raindrop.io Broken Links**：业界验证的标杆功能，展示对竞品的深度研究
2. **检测→保护闭环**：链接健康监控 + 页面存档 = 完整的知识保鲜体系
3. **误报过滤**：两次确认机制展示对工程可靠性的思考——"不是检测到就报警"
4. **三层通知递进**：从低调的卡片图标到紧急的邮件通知，展示 UX 分层设计思考
5. **调度架构复用**：与热度追踪（Feature D）共享优先级队列 + 速率限制器

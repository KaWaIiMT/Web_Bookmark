# 功能：AI 知识问答（RAG）

> 状态：规划完成 / 待开发

> 关联功能：[[AI自动分类]] [[关联推荐+学习路径]] [[书签详情]] [[页面永久存档]]

## 一、概述

为 MarkBox 新增基于 RAG（Retrieval-Augmented Generation）的个人知识库问答。用户用自然语言提问——"Rust 的 async/await 有哪些最佳实践？"——AI 从书签库中检索相关内容，综合多篇书签的摘要信息生成答案，附带引用来源，支持多轮追问。

这将是 MarkBox AI 能力深度的最高展示：从"能搜索"到"能回答"。已有 Embedding 向量基础设施可直接复用。

---

## 二、技术选型

### 决策 1：检索策略

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 纯向量检索 — 问题 Embedding → 余弦相似度 → Top-K 书签 → AI 生成答案 | 语义理解好，已有基础设施复用度高 | 精确关键词匹配可能遗漏（如"React 19" vs "React"） |
| **B** | 向量 + 关键词混合检索 — 向量召回 + 关键词精确匹配（标题/标签/摘要），两路召回合并 | 精确度更高，覆盖向量检索的盲区 | 实现复杂度增加，需要设计融合排序算法 |
| **C** | B + 多轮对话 + 引用追溯 — B 的基础上支持追问（"第一篇提到的那个 crate 叫什么？"），每轮附带引用来源 | 最完整的 RAG 体验，可深入探索知识。引用追溯建立用户信任 | 需要会话管理、上下文窗口策略 |

> ✅ **推荐 C**。混合检索保证召回质量，多轮对话让用户能深入追问而非一次性问答，引用追溯是建立 AI 可信度的关键——"AI 不是瞎编，是基于你的收藏"。

> ❌ **排除 A**：纯向量检索在处理技术术语的精确匹配时可能遗漏。例如用户搜索"React 19"时，向量检索可能返回"React 18"的内容。
>
> ❌ **排除 B**：缺少多轮对话能力，用户无法追问"那第二篇呢？"，体验不完整。

---

### 决策 2：检索粒度

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 书签级 — 以整篇书签的 AI 摘要 + 标题 + 描述 + 标签作为检索单元 | 数据已有，不额外抓取。隐私合规，无版权风险 | 依赖 AI 摘要质量，摘要缺失时信息不足 |
| **B** | 内容级 — 爬取每篇书签的完整正文，分块（chunk）存储和检索 | 信息最完整，能回答细节问题（"那篇文章第三段说了什么？"） | 需要页面存档基础设施（见 Feature C）。版权敏感。存储量大幅增加 |
| **C** | 混合粒度 — 优先使用摘要，摘要不足时降级到全文检索 | 兼顾两者 | 系统复杂度高，两套检索管道维护成本大 |

> ✅ **推荐 A**。书签级检索利用已有的 AI 摘要 + 标题 + 描述 + 标签，数据已存在且质量可控。对于个人书签管理器的 RAG 场景，摘要级信息足以回答 90% 的问题（"Rust 最佳实践有哪些""异步编程怎么学"）。当 Feature C（页面存档）落地后，可升级为全文检索。

> ❌ **排除 B**：需要全文抓取和分块存储，版权和法律风险较高。对于个人项目，在 MVP 阶段引入全文检索是过度工程。
>
> ❌ **排除 C**：两套检索管道增加了不必要的维护成本，且书签级检索已能满足核心需求。

---

### 决策 3：对话模型选择

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | DeepSeek V4（复用现有 `deepseek.ts` 客户端） | 项目已集成，零额外依赖。中文理解好，性价比高 | 相比 GPT-4o/Claude 在某些推理任务上稍弱 |
| **B** | Claude API（Anthropic） | 推理能力强，引用标注准确 | 需要新增 API 依赖和费用管理，国内访问可能受限 |
| **C** | 本地模型（Ollama + Qwen/Llama） | 零 API 费用，数据完全隐私 | 需要用户本地部署 GPU，问答质量取决于硬件配置，部署门槛高 |

> ✅ **推荐 A**。DeepSeek V4 已在项目中集成，中文能力优秀。对于基于书签摘要的 RAG 场景（而非复杂推理），DeepSeek 的能力完全够用。零额外集成成本。

> ❌ **排除 B**：引入新 API 依赖增加维护成本和费用管理复杂度。且 Anthropic API 国内访问稳定性不如 DeepSeek。
>
> ❌ **排除 C**：本地部署门槛过高，不符合"个人书签管理器"的零配置定位。答辩场景也难以保证每个评委的机器都能跑本地模型。

### 决策 4：UI 交互形态

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 仅搜索栏嵌入 — 在现有搜索框中输入问题，搜索结果顶部显示 AI 回答 | 融入现有界面，零学习成本 | 无法多轮对话，无会话历史，功能受限 |
| **B** | 独立全屏页面 — 新建 `/chat` 路由，独立于现有视图 | 功能完整，沉浸式体验 | 与主站割裂，用户需要在书签浏览和问答之间频繁切换 |
| **C** | 对话框 + 侧栏历史（类似 ChatGPT） — 作为新视图类型（ViewType: "chat"），融入十大视图体系 | 功能完整且不割裂。可与其他视图无缝切换。侧栏保留会话历史 | 开发量最大，需要完整的会话管理系统 |

> ✅ **推荐 C**。作为新视图融入现有的 ViewType 体系（grid/gallery/discover/compare/dashboard/timeline/weekly/graph/activity + chat），用户可以在浏览书签和知识问答之间一键切换。对话框形态是用户最熟悉的心智模型。额外提供 `Ctrl+K` 全局快捷键弹出轻量版问答浮窗。

> ❌ **排除 A**：无法多轮对话，AI 回答被搜索结果列表挤压，体验不完整。
>
> ❌ **排除 B**：独立页面与主站割裂，用户从问答中看到引用后无法快速查看书签详情。

---

### 决策 5：上下文窗口策略

| 选项 | 方案 | 优势 | 劣势 |
|---|---|---|---|
| **A** | 固定窗口 — 每次只传当前问题 + Top-8 检索结果，不保留历史 | 实现简单，token 消耗可控 | 无法多轮对话，每次都是"失忆"的对话 |
| **B** | 动态窗口 — 保留最近 N 轮对话 + 当前检索结果，超出时自动截断最旧消息 | 支持多轮追问，token 消耗可控 | 需要合理的截断策略和摘要缓冲区 |
| **C** | 全量历史 — 保留完整会话历史 + 检索结果 | 最完整的上下文 | token 消耗随对话增长，长对话可能超出模型上下文限制 |

> ✅ **推荐 B**。保留最近 8 轮对话历史（约 4 组 Q&A），超出时对最早的消息做 AI 摘要压缩后保留。保证多轮对话能力的同时控制 token 消耗。对于 RAG 场景，最近的对话上下文通常比历史对话更重要。

> ❌ **排除 A**：无法多轮对话，用户每次追问都需要重新描述上下文，体验差。
>
> ❌ **排除 C**：个人书签管理器的问答场景可能涉及大量书签引用，全量保留会导致 token 快速增长，超出 DeepSeek V4 的上下文限制。

## 三、交互设计

### 3.1 入口

- **全局入口**：顶部导航栏 Q&A 图标（💬），点击切换至 Chat 视图
- **书签详情入口**：BookmarkDetailSheet 中点击「问 AI」→ 自动带入当前书签作为上下文
- **搜索栏入口**：搜索框中输入问题，结果页顶部显示「在知识库中回答...」快捷入口
- **快捷键**：`Ctrl+K` 打开全局问答浮窗（轻量版）

### 3.2 问答流程

```
用户输入问题（或从预置模板选择）
  → 检索阶段（~200ms）：
    - 向量召回：问题 Embedding → 余弦相似度 → Top-10 书签
    - 关键词召回：问题分词 → 匹配标题/标签/摘要 → Top-5 书签
    - RRF 融合去重 → Top-8 候选
  → 生成阶段（~2-4s）：
    - 候选书签摘要+标题+标签作为上下文注入 Prompt
    - AI 生成答案 + 标注引用序号
    - 流式返回（SSE，逐字出现）
  → 展示阶段：
    - 答案气泡渐入
    - 底部引用来源卡片行
    - 追问输入框
```

### 3.3 答案展示

- AI 生成的结构化回答（Markdown 渲染，支持列表、代码块、加粗）
- 内联引用标记 `[1]` `[2]`，悬停显示来源标题
- 底部「📎 参考来源 (N)」可展开段落：每项含书签标题、相关片段、收藏时间、[打开书签] 链接

### 3.4 预置问题模板

- 「总结这篇文章的核心观点」
- 「这些书签中共有哪些最佳实践？」
- 「这个主题有哪些争议/不同观点？」

## 四、技术实现

### 4.1 数据模型

```prisma
model ChatSession {
  id        String        @id @default(cuid())
  userId    String
  title     String        // 自动从首条问题生成
  messages  ChatMessage[]
  createdAt DateTime      @default(now())
  updatedAt DateTime      @updatedAt
  user      User          @relation(fields: [userId], references: [id], onDelete: Cascade)
}

model ChatMessage {
  id          String      @id @default(cuid())
  sessionId   String
  role        String      // user | assistant
  content     String      // Markdown
  citations   String?     // JSON: [{ bookmarkId, title, snippet, relevanceScore }]
  feedback    String?     // positive | negative | null
  createdAt   DateTime    @default(now())
  session     ChatSession @relation(fields: [sessionId], references: [id], onDelete: Cascade)
}
```

### 4.2 API 设计

| 端点 | 方法 | 说明 |

| `/api/chat/sessions` | GET | 会话列表 |

### 4.3 核心算法

```typescript
// RRF 融合排序
function reciprocalRankFusion(
  vectorResults: ScoredDoc[],
  keywordResults: ScoredDoc[],
  k: number = 60
): ScoredDoc[] {
  const scores = new Map<string, number>();
  for (const [i, doc] of vectorResults.entries())
    scores.set(doc.id, (scores.get(doc.id) || 0) + 1 / (k + i + 1));
  for (const [i, doc] of keywordResults.entries())
    scores.set(doc.id, (scores.get(doc.id) || 0) + 1 / (k + i + 1));
  return [...scores.entries()]
    .sort((a, b) => b[1] - a[1])
    .slice(0, 8);
}
```

## 六、答辩亮点

1. **RAG 架构**：从"关键词搜索"到"知识问答"的 AI 能力升级，展示对前沿 AI 范式的理解
2. **混合检索 + RRF 融合**：向量语义匹配 + 关键词精确匹配，展示搜索工程深度
3. **引用追溯**：AI 答案可溯源、可验证——"不是黑盒 AI，是基于你的收藏"
4. **基础设施复用**：Embedding 向量在关联推荐中已生成，RAG 零额外抓取成本
